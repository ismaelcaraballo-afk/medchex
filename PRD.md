## Executive Summary

MedPharmChex is a free, web-based medication interaction checker that helps everyday people — patients, caregivers, and students — understand whether their medications are safe to take together. Unlike clinical tools that require medical training to interpret, MedPharmChex presents findings in plain English, backed by real FDA adverse event data and NIH-maintained drug databases. Our AI component (Claude) explains what the data means in human terms, but critically, it never decides how dangerous a drug combination is — that judgment is always made by real regulatory data.

---

## Problem Statement

Adverse drug interactions are one of the most preventable causes of serious medical harm in the United States. According to FDA reports, they contribute to over **125,000 deaths** and **1.5 million hospitalizations** every year. The majority of these incidents involve common prescription medications — blood thinners, anti-inflammatories, blood pressure medications — that are widely prescribed to elderly and chronically ill patients.

The core problem is an information gap. A patient picking up a new prescription has no fast, accessible, trustworthy way to check whether it conflicts with something they're already taking. The tools that exist today fall into one of three failure modes:

1. **Clinical databases** (like Epocrates or Drugs.com) are designed for medical professionals. They use clinical terminology, assume baseline pharmacology knowledge, and provide lists of interactions without context about severity or what to actually do.

2. **Consumer-facing sites** often lack sourcing transparency. They show warnings without citing the data behind them, which makes it impossible to know whether a flag is serious or just legal boilerplate.

3. **AI chatbots** used directly for this purpose are actively dangerous. Large language models can hallucinate drug interaction data, and a false negative — being told a dangerous combination is safe — could be fatal.

MedPharmChex addresses all three failure modes by combining authoritative data sources (NIH RxNorm and FDA FAERS) with AI-generated plain-language explanation, while keeping the AI strictly in an interpretive role rather than a diagnostic one.

## Solution

MedPharmChex is a single-page web application where a user types in two to five medication names and receives:

1. **A severity rating** — SAFE, CAUTION, or DANGEROUS — derived from RxNorm's clinical language and FDA adverse event data
2. **A plain-English explanation** — generated by Claude AI, translating the clinical findings into something a patient or caregiver can act on
3. **Real-world context** — the number of adverse event reports in the FDA FAERS database for the drug combination, giving users a sense of how common these outcomes are

The application is intentionally simple. There is no login, no account, no subscription. A user arrives, enters medications, and gets an answer. The design prioritizes trust and clarity over feature depth.

A critical architectural decision underpins the entire product: **the AI does not determine severity.** The SAFE / CAUTION / DANGEROUS classification is computed deterministically from FDA and RxNorm data — the same data a pharmacist would consult. Claude receives that pre-classified result and explains it. This separation ensures that even if the AI model produces unexpected output, it cannot produce a false-negative safety classification.

## Target Users

**Primary:**
- **Patients on multiple medications** — particularly elderly individuals managing chronic conditions such as diabetes, hypertension, and cardiovascular disease, who may be taking 5+ medications daily
- **Caregivers** — family members or home health aides managing medications for a parent, spouse, or patient who may not be able to research this themselves

**Secondary:**
- **Pharmacy and nursing students** — using MedPharmChex as a learning and reference tool during clinical rotations
- **General public** — anyone who has just received a new prescription and wants a quick, trustworthy second opinion before their next pharmacist visit

## Core Features (MVP)

### 1. Drug Name Resolution
The user types a medication name — brand name or generic — into a text input. The application queries the NIH RxNorm API to resolve the name to a standardized RxCUI code (RxNorm Concept Unique Identifier). This is the numeric identifier used across all US clinical drug systems. Accepting brand and generic names interchangeably is important because many patients only know one name for their medication (e.g., "Advil" rather than "ibuprofen"). The application supports checking up to 5 drugs simultaneously.

### 2. Interaction Detection
Using the resolved RxCUI codes, the application queries the RxNorm Interaction API, which returns all known interaction pairs for the given combination. RxNorm is maintained by the National Library of Medicine (NLM) and is the same system used in hospital EHR platforms and pharmacy management systems. It provides structured interaction data including clinical descriptions and severity language.

### 3. Severity Scoring Engine
The backend parses RxNorm's own clinical severity language to classify each interaction as one of three levels:
- **DANGEROUS** — triggered by keywords including "contraindicated," "fatal," "serious," "death"
- **CAUTION** — triggered by "moderate," "monitor closely," "may increase," "caution"
- **SAFE** — default if no flagged language is present

This logic is deterministic and fully auditable. The same drug pair will always return the same score. No randomness, no model drift, no hallucination risk. The overall severity for a drug combination is the highest severity found across all interaction pairs.

### 4. AI Plain-English Explanation
Once severity is determined by the scoring engine, the interaction data and severity level are passed to Claude (claude-sonnet-4-6). Claude's role is strictly interpretive: given this FDA-derived data and this severity classification, explain to a non-medical person what it means in plain language, what could happen if they take this combination, and what they should do next. Claude cannot override or change the severity classification — it only explains the one assigned by the data.

### 5. FDA FAERS Adverse Event Data
The application queries the OpenFDA FAERS (FDA Adverse Event Reporting System) database to retrieve the number of real-world adverse event reports associated with the primary drug being checked. This contextualizes the interaction result — a drug with 50,000 adverse event reports carries a different weight than one with 12.

## Technical Architecture

```
User Browser
     │
     ▼
React Frontend  (Vite + TypeScript + TailwindCSS + Framer Motion)
     │
     ▼  HTTP requests to /api/*
Express Backend  (Node.js, port 3001)
     │
     ├──▶  NIH RxNorm API        — drug name resolution, interaction detection
     ├──▶  OpenFDA FAERS API     — real-world adverse event reports
     ├──▶  Anthropic Claude API  — plain-English explanation (explain only)
     └──▶  node-cache            — 1-hour TTL response caching
```

The backend acts as a secure proxy for all external API calls. The Anthropic API key and any future credentials never touch the client. The frontend only ever communicates with our own Express server.

**Frontend:** React 19 · TypeScript · Vite · TailwindCSS · Framer Motion · Recharts
**Backend:** Node.js · Express 5 · express-rate-limit · node-cache
**External APIs:** NIH RxNorm · OpenFDA FAERS · Anthropic Claude (claude-sonnet-4-6)
**Deployment:** Vercel (frontend + serverless API routes)

## Data Sources

| Source | What It Provides | Why We Chose It |
|--------|-----------------|-----------------|
| **NIH RxNorm** | Drug name → RxCUI normalization, known interaction pairs with clinical descriptions | Official US drug vocabulary maintained by the National Library of Medicine. Free, no API key required, used in clinical EHR systems. The authoritative source for drug identification and interaction data. |
| **OpenFDA FAERS** | Real-world adverse event report counts for specific drugs | 15+ million voluntary adverse event reports submitted by patients, physicians, and pharmacists to the FDA. Provides real-world frequency context that clinical databases don't offer. |
| **Anthropic Claude** | Plain-English explanations of interaction data | Best-in-class language model for translating clinical findings into accessible language. Used only for explanation — never for classification or detection. |

## What We're NOT Building (MVP Scope)

The following are intentional exclusions from the MVP. They may be addressed in a post-capstone version but are out of scope for March 18:

- **User accounts or saved history** — adds authentication complexity without improving the core use case
- **Dosage-specific interaction analysis** — requires clinical-grade data and pharmacokinetic modeling beyond our current data sources
- **Native mobile app** — web-responsive design serves mobile users adequately for MVP
- **Real-time prescription import** — would require EHR integrations (FHIR/HL7) that are out of scope
- **Multi-language support** — English only for MVP

## Success Criteria

The following criteria define a successful MVP for Demo Day:

- [ ] A user can enter two or more drug names and receive a complete result (severity + explanation + FAERS count) in under 5 seconds on a standard connection
- [ ] The application handles misspelled or unknown drug names gracefully with a clear error message rather than crashing
- [ ] The Warfarin + Ibuprofen test case returns a DANGEROUS rating and an accurate AI explanation every time
- [ ] The application runs end-to-end with DEMO_MODE disabled, using live NIH and FDA APIs
- [ ] The backend starts with a clear error if required environment variables are missing
- [ ] The frontend displays meaningful error messages when the backend is unreachable, rather than showing a blank screen
